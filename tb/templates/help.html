📖 What the DUT does

avalon_mm_width_adapter bridges Avalon‑MM master and slave interfaces that have different data widths. A single RTL entity covers three operating modes, selected automatically at synthesis time by the generic values:

Upsize

The slave-side data bus is narrower than the master side (S_DATA_WIDTH < M_DATA_WIDTH). The adapter accumulates or fans-out narrow slave transactions into wider master transactions. Each slave word occupies one slot in the wider master word. Only a single master transaction is issued per slave transaction — the slot index derived from the slave address determines which byte lanes are active.

Downsize

The slave-side bus is wider than the master side (S_DATA_WIDTH > M_DATA_WIDTH). One slave transaction is split into multiple sequential master transactions, one per narrow slot. Read responses are reassembled before the slave sees them.

Pass-through

Equal widths (S_DATA_WIDTH = M_DATA_WIDTH). All signals are wired directly through with zero additional latency or logic.

🔢 Slot addressing

When the master bus is wider than the slave bus, the master word is divided into equally-sized slots. The slot index selects which portion of the master data bus a slave transaction maps to.

The slot index is extracted from the lower bits of the slave address — specifically the bits that would be zero on an aligned master address:

-- Example: S_DATA_WIDTH=32, M_DATA_WIDTH=64 → 2 slots, slot_bits = 1 slave address 0x0000_0000 → slot 0 → master[31:0] (byte-enables [3:0]) slave address 0x0000_0004 → slot 1 → master[63:32] (byte-enables [7:4]) -- Example: S_DATA_WIDTH=32, M_DATA_WIDTH=128 → 4 slots, slot_bits = 2 slave address 0x0000_0000 → slot 0 → master[31:0] be[3:0] slave address 0x0000_0004 → slot 1 → master[63:32] be[7:4] slave address 0x0000_0008 → slot 2 → master[95:64] be[11:8] slave address 0x0000_000C → slot 3 → master[127:96] be[15:12]
Note: The test bench currently exercises slot 0 and slot 1. For a 32→128 upsize these are the same two tests as 32→64 upsizing; the master bus is simply wider while the slot mapping logic is identical.

⚙️ VHDL Generics

📐 Valid configurations

All three rules must hold simultaneously:

Rule 1 — Divisibility

Both S_DATA_WIDTH and M_DATA_WIDTH must each be an exact positive integer multiple of SYMBOL_WIDTH:

S_DATA_WIDTH mod SYMBOL_WIDTH = 0 -- e.g. 32 mod 8 = 0 ✓ M_DATA_WIDTH mod SYMBOL_WIDTH = 0 -- e.g. 64 mod 8 = 0 ✓

Rule 2 — Power-of-two ratio

The ratio between the wider and narrower data width must be an exact power of two. Ratio = max(S,M) / min(S,M).

Valid: 32→64 (ratio 2) 32→128 (ratio 4) 64→32 (ratio 2) Invalid: 32→96 (ratio 3) 32→48 (ratio 1.5)

Rule 3 — Minimum width

Each data width must be at least SYMBOL_WIDTH bits wide.

Pre-defined configurations in the test runner (32→64, 32→128, 64→32, 32→32) all satisfy these rules. The Custom Widths mode validates them live in the UI before allowing a run.

⬆️ Upsize test cases

These seven tests apply to both the 32→64 and 32→128 configurations. In both cases the slave is 32 bits wide; the master bus width changes but the slot addressing logic is identical.

Write tests Write path

▶ test_upsize_write_slot0 Write into slot 0

Issues a 32-bit slave write to address 0x0000_0000. Because the address lower bits are zero, the slot index is 0, meaning the slave data must land in the lowest 32-bit lane of the master word.

What is verified:

  • The adapter issues exactly one master write transaction.
  • Master m_writedata[31:0] equals the slave s_writedata.
  • Master m_byteenable[3:0] equals the slave s_byteenable[3:0]; the upper byte-enable lanes are zero.
  • Slave s_waitrequest is deasserted after the master accepts the beat.
▶ test_upsize_write_slot1 Write into slot 1

Issues a 32-bit slave write to address 0x0000_0004 (one slave-word offset). The slot index becomes 1, so data must appear in the second 32-bit lane of the master word.

What is verified:

  • Master m_writedata[63:32] (or [63:32] for 32→64) equals s_writedata.
  • Master byte-enables are shifted: only m_byteenable[7:4] (32→64) or the corresponding lane (32→128) is active.
  • Lower byte-enable and data lanes are zero.

Read tests Read path

▶ test_upsize_read_slot0 Read from slot 0

Issues a 32-bit slave read to address 0x0000_0000 (slot 0). The adapter must issue one master read, wait for m_readdatavalid, then extract the lower 32 bits and present them on s_readdata.

What is verified:

  • Exactly one master read transaction is issued (m_read pulses once).
  • s_readdata equals m_readdata[31:0].
  • s_readdatavalid is asserted exactly when the master response arrives.
  • The slave sees no spurious waitrequest after the response.
▶ test_upsize_read_slot1 Read from slot 1

Issues a 32-bit slave read to address 0x0000_0004 (slot 1). The adapter must extract the second 32-bit lane from the master's wider read response.

What is verified:

  • s_readdata equals m_readdata[63:32] (32→64) or m_readdata[63:32] (32→128 slot 1).
  • Only the correct lane is extracted; no data from other slots leaks through.

Special cases Protocol edge cases

▶ test_upsize_byteenable Partial byte-enables

Issues a write with a sparse byte-enable mask (e.g. s_byteenable = 4'b0101 — only bytes 0 and 2 enabled). The adapter must position this mask correctly within the wider master byte-enable vector for the target slot.

What is verified:

  • Only the enabled slave bytes are marked active in the master byte-enable.
  • Other master byte-enable bits remain zero regardless of slot position.
  • Data in the disabled bytes is not required to be any specific value.
▶ test_upsize_waitrequest Master backpressure

A slave write is issued while the master memory model asserts m_waitrequest for one clock cycle before accepting the beat. The adapter must hold the transfer valid while backpressure is active.

What is verified:

  • The adapter keeps m_write, m_address, m_writedata, and m_byteenable stable during backpressure.
  • s_waitrequest is asserted to the slave while waiting, then released once the master accepts.
  • The write data is not corrupted or lost.
▶ test_upsize_pipelined_reads Pipelined outstanding reads

Issues two back-to-back slave reads targeting different slots before the first master response returns. This exercises the slot-tracking FIFO (us_idx_fifo) inside the adapter that records which lane to extract for each pending read.

What is verified:

  • The adapter can accept a second slave read while the first master response is still outstanding.
  • Responses are returned in order: slot-0 data appears on s_readdata first, then slot-1 data on the following valid cycle.
  • No read response is dropped or assigned to the wrong slave transaction.
Why a FIFO? Because the master bus is pipelined (multiple reads can be outstanding), the adapter cannot derive the slot index from the master response alone — it must remember the order in which reads were issued. The FIFO stores one slot index entry per in-flight read.

⬇️ Downsize test cases

These four tests use the 64→32 configuration: the slave sees a 64-bit bus, the master is 32 bits. One slave transaction is split into two sequential master transactions (beats).

▶ test_downsize_write 64-bit write split into two 32-bit writes

A single 64-bit slave write is decomposed into two sequential 32-bit master writes. Beat 0 carries the lower 32 bits to the base address; beat 1 carries the upper 32 bits to base address + 4.

What is verified:

  • The adapter asserts s_waitrequest while both master beats are issued.
  • Beat 0: m_address = s_address, m_writedata = s_writedata[31:0], m_byteenable = s_byteenable[3:0].
  • Beat 1: m_address = s_address + 4, upper half of data and byte-enables.
  • s_waitrequest is released only after the second master beat completes.
▶ test_downsize_read Two 32-bit reads reassembled into one 64-bit response

A single 64-bit slave read triggers two 32-bit master reads. The two responses are concatenated and presented as one 64-bit slave read response.

What is verified:

  • Two master reads are issued to consecutive addresses.
  • s_readdata[31:0] equals the first master response.
  • s_readdata[63:32] equals the second master response.
  • s_readdatavalid is asserted only after both responses are received.
▶ test_downsize_partial_be Sparse byte-enables suppress sub-beats

A 64-bit slave write is issued with byte-enables that only activate bytes in the upper half (e.g. s_byteenable = 8'b1111_0000). The adapter should suppress the first master write beat entirely and only issue the second beat.

What is verified:

  • Only one master write is issued (the upper beat with non-zero byte-enables).
  • No spurious master write to the lower address occurs.
  • The slave receives its acknowledgement after the single active beat completes.
▶ test_downsize_waitrequest Master backpressure during multi-beat sequence

The master asserts m_waitrequest during the first beat of a two-beat downsize write. The adapter must stall that beat and then resume the sequence correctly.

What is verified:

  • The first beat is held stable while m_waitrequest is high.
  • After backpressure releases, the first beat completes and the second beat is issued without loss.
  • The slave's s_waitrequest remains asserted until both beats are done.

➡️ Pass-through test cases

These three tests use the 32→32 configuration where slave and master widths match. No data transformation occurs; the adapter is purely combinational wiring.

▶ test_passthrough_write Write signals forwarded unchanged

A 32-bit slave write is presented at the slave port. All write signals must appear at the master port with zero additional latency.

What is verified:

  • m_address = s_address
  • m_writedata = s_writedata
  • m_byteenable = s_byteenable
  • m_write = s_write (combinational — no registered delay)
▶ test_passthrough_read Read response forwarded unchanged

A slave read request is forwarded to the master. The master's read response (m_readdata, m_readdatavalid) is wired directly back to the slave.

What is verified:

  • s_readdata = m_readdata when valid.
  • s_readdatavalid = m_readdatavalid (no buffering).
▶ test_passthrough_waitrequest Backpressure forwarded unchanged

The master asserts m_waitrequest for one cycle. In pass-through mode this signal must propagate directly to s_waitrequest with no additional registered delay.

What is verified:

  • s_waitrequest = m_waitrequest combinationally every clock cycle.
  • No spurious transactions are generated while backpressure is active.

🔧 Custom Widths mode

Custom Widths lets you supply arbitrary generic values at run time without editing any source files. The UI validates your inputs against the three rules in Valid configurations before enabling the Run button.

UI fieldMaps to genericDefaultNotes
S_DATA_WIDTH S_DATA_WIDTH 32 Slave bus width in bits.
M_DATA_WIDTH M_DATA_WIDTH 128 Master bus width in bits.
ADDR_WIDTH ADDR_WIDTH 32 Address bus width. Usually left at 32.
SYMBOL_WIDTH SYMBOL_WIDTH 8 Bits per byte-enable symbol. Usually left at 8.

The adapter mode (upsize / downsize / pass-through) and the matching test suite are inferred automatically from the S_DATA_WIDTH vs M_DATA_WIDTH comparison. The run.py script receives the values via environment variables (VUNIT_CUSTOM_S_WIDTH, VUNIT_CUSTOM_M_WIDTH, VUNIT_CUSTOM_ADDR_WIDTH, VUNIT_CUSTOM_SYM_WIDTH).

📈 Waveform viewer

Enable Show Waveforms before running tests. After the run finishes, a Waveforms card appears below the result banner.

How it works

When waveforms are enabled, the Flask server sets the environment variable VUNIT_WAVE_DIR before launching run.py. For each test configuration, run.py passes --vcd=<path> as a GHDL simulation flag. GHDL writes a standard 4-state VCD file per test.

The server parses the VCD at the /waves/data endpoint, returning JSON signal data. The browser renders it on an HTML Canvas using a custom WaveformViewer class.
